# 사용자 0명일 때 모니터링부터 깔았다 — 숫자야구 봇 CloudWatch 도입기

> 카카오 오픈빌더 스킬 서버(Kotlin + Spring Boot)를 EC2에 띄워둔 사이드 프로젝트에
> CloudWatch 알람을 붙인 이야기. 지표를 어떻게 골랐고, IaC 대신 왜 셸 스크립트를 택했으며,
> "지표는 오는데 로그만 0건"인 권한 버그를 어떻게 격리했는지, 그리고 "잘못된 유저 입력을
> 에러로 봐야 하나?"라는 질문이 로그 설계를 어떻게 바꿨는지에 대한 기록.

## 왜 사용자가 0명인데 모니터링부터?

부하 테스트를 하려던 참이었다. 그런데 부하를 걸어도 **그 부하가 서버에 무슨 일을 일으키는지
볼 눈이 없으면** 테스트가 아니라 그냥 요청 던지기다. 그래서 순서를 뒤집었다.

- 모니터링을 먼저 깔면, 이후 부하 테스트가 곧 이 알람/지표의 **실검증**이 된다.
- 출시 전(실사용자 0)이라 오탐이 나도 나 말고 아무도 안 깨운다 — 임계치를 마음껏 튜닝할
  가장 안전한 시점.
- 앱 코드 변경이 (거의) 없다. 100% 인프라 작업이라 게임 로직과 격리된다.

한 줄 요약: **관측 가능성은 트래픽이 오기 전에 세팅해야 트래픽이 왔을 때 의미가 있다.**

## 무엇을 감지할 것인가 — 지표 5개 고르기

"모니터링하자"는 막연하다. 구체화하려면 **어떤 신호가 뜨면 내가 실제로 행동할 것인가**를
먼저 정해야 했다. 행동으로 이어지지 않는 알람은 노이즈고, 노이즈는 알람 전체를 무시하게
만든다. 그 기준으로 5개만 남겼다.

| 알람 | 지표 | 무엇을 잡나 | 임계(시작값) |
|------|------|------------|-------------|
| 앱 에러 급증 | `Baseball/App/error_count` | 진짜 서버 장애 | 5분 합 > 10 |
| 앱 지연 | `Baseball/App/slow_count` | 카카오 5초 타임아웃 직전 | 5분 합 >= 3 |
| 메모리 | `CWAgent/mem_used_percent` | 2GB에 JVM 2개, OOM 전조 | 10분 평균 > 85% |
| CPU 크레딧 고갈 | `AWS/EC2/CPUCreditBalance` | 버스터블 스로틀 전조 | 10분 평균 < 60 |
| CPU 사용률 | `AWS/EC2/CPUUtilization` | 크레딧 고갈 전 추세 | 10분 평균 > 70% |

여기엔 이 프로젝트만의 제약이 녹아 있다.

- **카카오 5초 타임아웃.** 스킬 서버는 5초 안에 응답 못 하면 사용자에게 그냥 실패로
  보인다. 그래서 지연은 500 에러가 아니라 **3초(`SLOW_THRESHOLD_MS`) 초과**를 위험 신호로
  잡는다. 3초를 넘기 시작하면 5초 절벽이 코앞이다.
- **버스터블 인스턴스(t4g.small).** CPU 사용률만 보면 함정이다. 버스터블은 크레딧이
  마르면 사용률이 낮아도 baseline으로 스로틀된다. 그래서 `CPUUtilization`(추세)과
  `CPUCreditBalance`(잔량)를 **둘 다** 본다. 크레딧 잔량 하락은 "m 계열로 갈아탈 때"라는
  신호다.
- **한 인스턴스에 prod + dev JVM 2개.** 스테이징을 프로덕션과 같은 EC2 호스트에 얹어
  비용을 아꼈더니, 메모리 2GB를 둘이 나눠 쓴다. 그래서 메모리 알람 임계가 공격적(85%)이다.

### 트레이드오프: 임계치는 전부 "추측"이다

솔직히 말하면 이 숫자들(10, 3, 85%, 60, 70%)은 **데이터가 아니라 추측**이다. 실사용자가
0명이니 베이스라인이 없다. 두 가지 선택지가 있었다.

1. 베이스라인이 쌓일 때까지 알람을 안 건다.
2. 보수적인 시작값으로 일단 걸고, 트래픽이 오면 튜닝한다.

2번을 택했다. **알람이 없는 것보다 오탐 나는 알람이 낫다** — 없으면 장애를 놓치지만,
오탐은 임계치만 올리면 된다. 대신 모든 문서와 스크립트 주석에 "이건 시작값이다"라고
못박아서, 나중의 내가 이 숫자를 신성한 것으로 착각하지 않게 했다. 모든 알람에
`TreatMissingData=notBreaching`를 붙인 것도 같은 맥락 — 트래픽이 없어 데이터가 없는 상태를
"정상"으로 취급해 조용한 새벽에 안 울리게 했다.

## 접근 방식 트레이드오프 — Terraform vs 콘솔 vs 셸 스크립트

세 가지 갈림길이었다.

- **콘솔 클릭.** 제일 빠르지만 **재현 불가**. 반년 뒤 "이 알람 왜 이렇게 설정했지"에 답할
  기록이 없다. 실수로 지우면 복구할 수 없다.
- **Terraform(IaC).** 정석. 하지만 EC2 1대 + RDS 1대짜리 솔로 프로젝트에 상태 파일 관리,
  provider 버전, `terraform apply` 파이프라인까지 얹는 건 **과하다**. 도구 유지보수가 감시
  대상보다 무거워지는 순간이다.
- **멱등 셸 스크립트 + README.** 중간 지점. AWS CLI 호출을 `set -euo pipefail` 스크립트
  하나에 모아 버전 관리하고, 값은 환경변수로 주입한다.

셋째를 택했다. 핵심은 **멱등성** — 몇 번을 돌려도 같은 결과라야 "재설정 스크립트"가 되지
"위험한 일회성 명령"이 안 된다.

- `sns create-topic` — 같은 이름이면 같은 ARN 반환(멱등).
- `put-metric-filter` / `put-metric-alarm` — upsert(멱등).
- `--dry-run` 플래그 — 실제 생성 대신 실행할 CLI를 출력. 리뷰하고 나서 진짜로 돌린다.

```bash
# 리뷰: 무슨 CLI가 실행될지 먼저 본다
ALARM_EMAIL=you@example.com INSTANCE_ID=i-xxxx ./setup-cloudwatch.sh --dry-run

# 실행
ALARM_EMAIL=you@example.com INSTANCE_ID=i-xxxx ./setup-cloudwatch.sh
```

### 이메일 주소는 왜 환경변수인가 — 작은 보안 결정

알람 수신 이메일을 스크립트에 하드코딩하고 싶은 유혹이 있었다. 안 했다.

- 커밋된 파일에 개인 이메일이 박히면 **공개 레포에 영원히 남는다**(삭제해도 git 히스토리엔
  남는다).
- 그래서 `ALARM_EMAIL`을 **필수 환경변수**로 만들고, 미지정 시 usage 출력 후 즉시 종료
  시켰다. `INSTANCE_ID`도 마찬가지 — 실수로 엉뚱한 인스턴스에 알람을 걸지 않도록 기본값을
  안 줬다.

이건 이 프로젝트가 DB 접속정보를 레포에 안 넣고 환경변수로 주입하는 원칙과 정확히 같은
선상이다. 비밀은 코드 옆이 아니라 실행 환경에 둔다.

## 가장 많이 배운 버그 — "지표는 오는데 로그만 0건"

셋업을 끝내고 확인하는데 이상했다.

- CloudWatch Agent: `running`.
- 메모리·디스크 지표(`CWAgent` 네임스페이스): **정상 수집됨**.
- IAM 역할, config 파일: 다 정상.
- 그런데 로그 그룹 `/baseball/app`에 이벤트가 **0건**. `describe-log-streams`가 빈
  배열(`[]`)을 반환했다. 스트림이 아예 안 만들어졌다는 뜻(스트림은 첫 성공적인
  put-log-events 때 lazy하게 생긴다).

### 격리: 지표가 정상이라는 게 결정적 단서였다

여기서 CloudWatch Agent의 구조를 알면 범위가 확 좁혀진다. **지표 파이프라인과 로그
파이프라인은 독립적**이다.

- 지표 수집: 시스템 값을 읽어 API로 push. 파일 접근 불필요.
- 로그 수집: 지정된 파일을 **tail(읽기)** 해서 push.

지표가 멀쩡히 온다 = Agent 프로세스도, IAM 권한도, 네트워크도 다 정상이다. 그렇다면 로그만
죽은 이유는 **하나뿐** — tail할 파일을 못 읽는다. 문제를 "AWS 어딘가"에서 "로컬 파일 읽기
권한"으로 격리한 순간이었다.

### 진범: 부모 디렉터리의 실행권한

```
/home/ubuntu           0750  ubuntu:ubuntu
/home/ubuntu/logs/     ...
/home/ubuntu/logs/server.log   0644  (world-readable)
```

`server.log` 자체는 0644라 누구나 읽을 수 있다. 그런데 로그가 안 갔다. 왜?

**CloudWatch Agent는 root가 아니라 `cwagent` 유저로 돈다.** 그리고 리눅스에서 파일 하나를
읽으려면 그 파일까지 가는 **모든 상위 디렉터리에 실행권한(x)** 이 있어야 한다.
`/home/ubuntu`가 `0750`이고 소유자가 `ubuntu:ubuntu`라, `other`에겐 아무 권한이 없다.
`cwagent`는 `ubuntu` 그룹에 속하지도 않았다. 결국 `cwagent`는 `/home/ubuntu`를
**통과(traverse)조차 못 해서** 그 안의 world-readable 파일에 도달하지 못했다.

파일 권한을 아무리 열어도 소용없다. 병목은 파일이 아니라 **그 위를 지나가는 길**이었다.

```bash
# 재현/확인: cwagent 입장에서 파일이 읽히나?
sudo -u cwagent head /home/ubuntu/logs/server.log   # Permission denied

# 수정: cwagent를 ubuntu 그룹에 넣어 traverse 권한 부여
sudo usermod -aG ubuntu cwagent
# 그룹 반영은 새 프로세스부터 → Agent 재기동 필수
sudo systemctl restart amazon-cloudwatch-agent   # (또는 agent-ctl)

# 검증
sudo -u cwagent head /home/ubuntu/logs/server.log   # 읽힘
# 재기동 후 스트림 i-0d131fd9dd2bc8eee 생성, 최신 타임스탬프 로그 유입 확인
```

### 왜 이 수정이 영구적인가

배포 스크립트가 `server.log`를 새로 만들어도 **디렉터리 권한과 그룹 멤버십은 유지**된다.
그래서 재배포 때마다 다시 손댈 필요가 없다. 파일 권한에 의존하는 수정이었다면 배포마다
깨졌을 것이다 — 부모 디렉터리 레벨에서 고친 게 그래서 옳았다.

**교훈:** 파일을 못 읽으면 파일 권한부터 보게 되지만, 진짜 답은 종종 경로 위쪽에 있다.
그리고 "A는 되는데 B만 안 된다"(지표 O, 로그 X)는 항상 최고의 디버깅 단서다 — 둘의
**차이**가 곧 원인이다.

## 로그 설계 논쟁 — 잘못된 유저 입력은 "에러"인가?

알람의 핵심인 `error_count`는 로그에서 뽑는다. Metric Filter가 로그 그룹을 훑어
`status=ERROR` 패턴이 보이면 카운트를 1 올린다. 그래서 **무엇을 ERROR로 로깅하느냐**가
알람의 품질을 통째로 결정한다.

여기서 스스로에게 던진 질문이 설계를 바꿨다.

> "자릿수를 틀리게 제출하거나, 문자로 입력한 경우도 에러 로그로 봐야 하나?"

처음 구현은 `try/catch`에서 잡히는 모든 예외를 `status=ERROR`로 남겼다. 그런데
숫자야구에서 **잘못된 입력은 지극히 정상적인 유저 흐름**이다.

- "12"만 입력 → 자릿수 부족(`IllegalArgumentException`)
- "abcd" 입력 → 유효하지 않은 문자(`IllegalArgumentException`)
- 게임 시작 안 하고 추측 → "진행 중인 게임 없음"(`IllegalStateException`)

이걸 전부 ERROR로 세면, **사용자가 게임을 정상적으로 하다 오타 한 번 낼 때마다 에러
카운트가 오른다.** 그러면 `error_count`는 "서버가 아픈 정도"가 아니라 "사람들이 오타 내는
빈도"를 재게 되고, 알람은 의미를 잃는다.

### 해결: REJECTED / ERROR 분리

`LogTraceAspect`의 catch 블록에서 예외 종류로 상태를 갈랐다.

```kotlin
status = if (e is IllegalArgumentException || e is IllegalStateException) {
    "REJECTED(${e.javaClass.simpleName})"   // 정상 유저 흐름 — 안내로 처리됨
} else {
    "ERROR(${e.javaClass.simpleName})"      // 진짜 서버 장애
}
```

- `REJECTED` — 예상된 입력/상태 예외. advice가 친절한 안내 메시지로 바꿔 응답한다.
  로그 레벨도 ERROR가 아니라 INFO다.
- `ERROR` — 예상 못 한 모든 것(DB 다운, NPE 등). 이것만 ERROR 레벨로 남긴다.

Metric Filter 패턴은 `"status=ERROR"`. 이제 `error_count`는 **진짜 서버 장애만** 센다.
사용자가 아무리 오타를 내도 알람은 조용하다. 예외를 두 종류로 나눈 게 아니라, **"이건
행동이 필요한 신호인가, 정상적인 하루인가"** 를 코드가 구분하게 만든 것이다.

이 결정 하나가 없었다면 `error_count > 10` 알람은 출시 첫날 유저 몇 명의 오타로 울렸을
것이고, 나는 이틀 만에 그 알람을 꺼버렸을 것이다. 좋은 알람은 임계치 튜닝이 아니라 **무엇을
세느냐**에서 시작한다.

### 관측을 위한 원칙 하나: 예외를 삼키지 마라

`LogTraceAspect`는 `@Around`로 컨트롤러를 감싸 예외를 **로깅한 뒤 다시 던진다**. 삼키지
않는다. 이유는 명확하다 — aspect가 예외를 먹어버리면 관측은 되지만 다운스트림의
`@RestControllerAdvice`가 사용자용 안내 응답을 못 만든다. 관측과 처리는 다른 레이어의
일이다. aspect는 보기만 하고, 응답은 advice가 만든다. 그리고 응답시간 측정을 `finally`에
둬서, 요청이 중간에 죽어도(타임아웃·OOM) "들어온 요청"의 흔적과 소요시간은 남는다.

## 데이터 흐름 전체

```
앱(LogTraceAspect) → /home/ubuntu/logs/server.log
      → CloudWatch Agent(cwagent 유저) → 로그 그룹 /baseball/app
            → Metric Filter 2개 → 커스텀 지표 Baseball/App{error_count, slow_count}
CloudWatch Agent → CWAgent 네임스페이스 → mem_used_percent
EC2 기본 지표 → AWS/EC2 → CPUUtilization, CPUCreditBalance

5개 Alarm → 1개 SNS 토픽(baseball-alarms) → 이메일 구독
```

## 회고 — 무엇을 남겼나

- **관측을 트래픽보다 먼저.** 부하 테스트 전에 눈을 달았더니, 부하 테스트 자체가 알람의
  첫 실검증이 됐다.
- **도구는 감시 대상보다 가벼워야 한다.** Terraform이 정석이어도 EC2 1대엔 과하다. 멱등
  스크립트 + `--dry-run`이 재현성과 단순함의 균형점이었다.
- **"A는 되고 B만 안 된다"는 금광.** 지표 O / 로그 X의 대비가 문제를 AWS 전체에서 파일
  읽기 권한 한 곳으로 좁혔다. 그리고 파일 권한이 아니라 부모 디렉터리 traverse가 진범이었다.
- **좋은 알람은 임계치가 아니라 정의에서 나온다.** REJECTED/ERROR 분리 없이는 어떤
  임계치를 잡아도 `error_count`가 오타를 셌을 것이다.
- **임계치는 추측임을 인정하고 그렇게 표시하라.** 시작값이라고 문서에 못박아, 미래의 내가
  데이터로 교체할 여지를 남겼다.

### 아직 안 한 것 (YAGNI)

- 알람 실동작 검증(부하 테스트에서) 및 베이스라인 확보 후 임계치 튜닝 — 다음 스텝.
- RDS 알람(커넥션/스토리지), CloudWatch 대시보드, Micrometer/Prometheus — 필요해지면.

관측은 완성하는 게 아니라 트래픽과 함께 자라는 것이다. 지금은 씨앗을 심고, 첫 부하가 물을
줄 차례다.
